1.1. Conceitos

O seguinte texto foi extraído de Tanenbaum e Woodhull (2008).

Quando um programa de usuário executa a instrução n = read(fd, buffer, nbytes); para realizar a leitura sobre um arquivo normal, a função de biblioteca read é chamada com três parâmetros. A função constrói uma mensagem contendo esses parâmetros, junto com o código de read como o tipo da mensagem, e envia essa mensagem para o sistema de arquivos e bloqueia, esperando a resposta.

Quando a mensagem chega, o sistema de arquivos usa o tipo da mensagem como índice em suas tabelas para chamar a função que trata da leitura.

Essa função extrai o descritor de arquivo da mensagem e o utiliza para localizar a entrada flip e, depois, o i-node do arquivo a ser lido. Então a requisição é dividida em partes, de modo que cada parte caiba dentro de um bloco. Por exemplo, se a posição do arquivo corrente é 600 e foram solicitados 1024 bytes, a requisição será dividida em duas partes, de 600 a 1023 e de 1024 a 1623 (supondo blocos de 1 KB).

Para cada uma dessas partes, por sua vez, é feita uma verificação para ver se o bloco relevante está na cache. Se o bloco não estiver presente, o sistema de arquivos selecionará o buffer usado menos recentemente que não esteja em uso e o reinvindicará, enviando uma mensagem para o driver de dispositivo de disco para reescrevê-lo, caso esteja sujo. Então, o driver de disco é solicitado a buscar o bloco a ser lido.

Quando o bloco estiver na cache, o sistema de arquivos enviará uma mensagem para a tarefa de sistema pedindo para que ela copie os dados no lugar apropriado no buffer do usuário (isto é, os bytes de 600 a 1023 no início do buffer e os bytes 1024 a 1623 no deslocamento 424 dentro do buffer). Após a cópia ser feita, o sistema de arquivos envia uma mensagem de resposta para o usuário, especificando quantos bytes foram copiados.

Quando a resposta volta para o usuário, a função de biblioteca read extrai o código da resposta e o retorna como o valor da função para o processo que fez a chamada.

Ambas as chamadas ao sistema realizam operações sequenciais no disco, de acordo com o ponteiro do arquivo e ambas retornam como resultado o número de bytes manipulados na operação ou um código de erro.

2. Testando Chamadas ao Sistema Relacionadas com Arquivos

Deseja-se a implementação de um programa que utilize as chamadas open, seek, read, write e close para, a partir de parâmetros passados na linha de comando pelo usuário, seja possível escolher uma sequência de chamadas ao sistema. Por exemplo, open, seek e read; ou open, write, seek, write e close. A sequência deve ser executada a partir de um único pedido na linha de comando.

Procure saber quais são os parâmetros das chamadas open, seek, read, write e close. Em seguida, projete uma maneira de passar os parâmetros ao programa.

Teste e relate os resultados das seguintes sequências:

·         write 12 caracteres (aaabcdefghij), seek para 0 (zero) e write de 2 caracteres (##);

·         open (arquivo novo novo) write 12 caracteres (aaabcdefghij), seek para 0 (zero) e write de 2 caracteres (##) close;

·         open (arquivo novo novo1) write 12 caracteres (aaabcdefghij), seek para 100 (cem) e write de 2 caracteres (##) close;

·         open (arquivo novo novo2) write 12 caracteres (aaabcdefghij), seek para 100000 (cem mil) e write de 2 caracteres (##) seek para 100001 (cem mil e um) write de 5 caracteres (@@@@@)

·         open (arquivo leitura escrita novo2) seek para 50000 (cinquenta mil) e read de 2 caracteres, seek para 100003 (cem mil e três) e write os 2 caracteres lidos

Use stat, readlink após cada execução do programa, documente o resultado explicando-o.
